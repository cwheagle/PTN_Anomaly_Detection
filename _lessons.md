# PTN 이상탐지 프로젝트 학습 레슨 (_lessons.md)

이 파일은 개발 과정에서 발생한 시행착오와 해결책을 기록하여, 향후 유사 프로젝트나 운영 단계에서 동일한 실수를 방지하기 위한 지식 베이스입니다.

## 1. 데이터 스키마 및 일관성 (Data Consistency)
- **교훈:** DB 컬럼명(`ip_addr`, `cid`, `lid`)과 코드 내 변수명(`equipment_id`, `port_id`)이 혼용될 경우 추론 및 결과 매핑 단계에서 반드시 오류가 발생함.
- **원칙:** 전 계층(DB -> Preprocess -> Model -> Inference -> Save)에서 원본 DB의 식별자 명칭을 최우선으로 유지하며, 변환이 필요한 경우 별도의 매핑 레이어를 명시적으로 둠.

## 2. 시계열 데이터 정합성 (Time-Series Integrity)
- **교훈:** 15분 단위 데이터에서 초 단위 오차나 누락이 발생하면 LSTM 시퀀스(Window)가 밀리거나 잘못된 시점의 이상을 탐지하게 됨.
- **해결:** 
    - `pd.to_datetime().dt.round('1min')`을 통해 시간 정규화(Rounding)를 필수적으로 수행.
    - 시퀀스 생성 시 시간 간격(예: 30분 이상)을 체크하여 데이터 단절 시 시퀀스를 분리/스킵하는 로직을 구현하여 인덱스 정합성 확보.

## 3. 모델 차원 관리 (Dimensionality Management)
- **교훈:** `config.py`의 `input_dim`과 `DataProcessor`의 `feature_cols` 개수가 일치하지 않으면 런타임에 텐서 차원 오류가 발생함. (예: 초기 설계 8개 vs 최종 구현 5개)
- **원칙:** 피처 추가/삭제 시 `config.py`를 중앙 제어판으로 사용하고, 모든 테스트 코드에서 이를 동적으로 참조하도록 설계함.

## 4. 메모리 및 성능 최적화 (Resource Management)
- **교훈:** 대량의 데이터를 한꺼번에 로드하여 학습하거나 추론할 때 메모리 사용량이 급증하여 프로세스가 강제 종료(OOM)될 수 있음.
- **해결:** 
    - 학습 시 `DataLoader`를 사용하여 배치 단위로 처리.
    - 추론 시에도 장비별/포트별 그룹화 처리를 통해 메모리 점유율을 분산시킴.

## 6. 시계열 정합성 및 데이터 정제 (Advanced Time-Series)
- **교훈**: DB 행(Row) 누락 시 시간축 재구성(Reindexing)이 없으면 윈도우 정합성이 깨져 장기간 추론이 중단됨.
- **해결**: 15분 단위 강제 재인덱싱과 `limit=1` 보간을 결합하여, 단일 누락은 복구하고 대량 누락은 안전하게 버리는 가용성 확보.
- **교훈**: 스케일러(Fit) 학습 시 결측치를 특정 값으로 채우면 통계적 평균/표준편차가 왜곡됨.
- **원칙**: 반드시 `dropna()` 후 순수 데이터로만 스케일러를 학습시켜 '진짜 정상'의 분포를 유지해야 함.
- **교훈**: 보간되지 않은 결측치가 윈도우에 섞여 들어가면 모델이 비정상적인 예측을 수행함.
- **해결**: `is_nan_mask`를 활용하여 결측치가 하나라도 포함된 12개 시점의 윈도우는 통째로 폐기(`Skip`)하는 로직이 오탐 방지의 핵심임.

## 8. 극단적 이상치와 로그 변환 (Handling Extreme Outliers)
- **교훈**: 10의 19승(1e19)과 같이 물리적으로 불가능한 수치는 스케일러만으로 해결되지 않으며, 모델 계산 시 에러(Infinity)를 유발함.
- **해결**: 
    - **사전 필터링**: 학습 단계에서 `1e9` 이상의 수치가 발견되는 포트는 오염된 데이터로 간주하여 통째로 드랍(`_filter_low_quality`).
    - **로그 스케일**: 트래픽 데이터의 거대한 편차를 `log1p`로 압축하여 모델이 상대적 변화를 학습하게 유도.
    - **RobustScaler**: 평균/표준편차 대신 중앙값/사분위수를 사용하여 이상치에 의한 스케일 왜곡 방지.
- **원칙**: 로그 변환된 데이터에는 MSE(Mean Squared Error)를 사용하여 상대적 비율 오차를 효과적으로 포착함.

## 9. DB 스키마 및 저장 정합성 (Schema Sync)
- **교훈**: `inference.py`에서 산출되는 컬럼(예: 23개)과 DB의 컬럼 구성, 그리고 `save_results`의 INSERT 쿼리 내 바인딩 순서가 100% 일치해야 함. 필드 하나가 누락되거나 순서가 바뀌면 저장 단계에서 전체 배치 작업이 실패함.
- **해결**: 필드 추가 시 [Inference(표준 컬럼) -> DB Schema(테이블 생성 쿼리) -> DB Insert(바인딩 순서)]의 3단계를 동시에 업데이트하고, 항상 테이블을 재 생성하여 정합성을 확인해야 함.

## 10. 예지 정비를 위한 추세 분석 범위 (Trend Window)
- **교훈**: 윈도우 기반 모델에서 '현재 상태'가 아닌 '추세(Slope)'를 계산하려면, 최소 2개 이상의 점수가 필요하며 이를 위해 `window_size`보다 큰 범위의 데이터 조회가 필요함.
- **해결**: `FETCH_MINUTES = (window_size + 4) * INTERVAL` 공식을 사용하여, 한 번의 추론에서 현재 점수와 과거 1시간의 변화율을 동시에 산출할 수 있는 여유 범위를 확보함.

## 11. 스케줄러 로그 관리 (Log Hygiene)
- **교훈**: pandas의 `read_sql` 등에서 발생하는 `UserWarning`은 시스템 운영에 지장이 없으나, 15분 주기로 가동되는 로그를 오염시켜 실제 에러 식별을 방해함.
- **원칙**: 운영 단계의 스케줄러에서는 권고성 경고(UserWarning)를 명시적으로 무시(`warnings.filterwarnings`) 처리하여 가독성을 확보함.

## 12. 실시간 저장 vs 백테스팅 분석 분리 (Real-time vs Historical)
- **교훈**: 추세 분석을 위해 다중 윈도우를 분석할 때, 모든 결과를 반환하면 DB에 중복 데이터가 쌓이게 됨. 반대로 최신 1건만 반환하도록 고정하면 과거 데이터를 통한 성능 검증(Backtesting)이 불가능해짐.
- **해결**: `detect()` 함수에 `latest_only` 플래그를 도입. 스케줄러는 `True`(최신 1건)로 설정하여 DB 정합성을 유지하고, 테스트 코드는 `False`(전체 결과)로 설정하여 과거 장애 패턴에 대한 전수 조사가 가능하도록 설계함.

## 13. DB 테이블 유실 및 복구 (Disaster Recovery)
- **교훈**: 운영 중 실수나 작업으로 DB 테이블(`anomaly_detection`)이 삭제되더라도, 시스템의 `DBConnector`가 초기화 시 `CREATE TABLE IF NOT EXISTS` 로직을 포함하고 있어 즉시 복구가 가능함.
- **원칙**: 중요한 탐지 결과는 DB 외에도 일자별 CSV 파일(`data/history_YYYYMMDD.csv`)에 이중 기록(Dual Logging)하여 데이터 유실에 대비함.

## 14. 최신 시각 기반 결과 필터링 (Latest Batch Filtering)
- **교훈**: `latest_only=True` 상황에서 `groupby().tail(1)`을 사용하면, 특정 포트의 데이터 수집이 지연되어 과거(예: 45분 전) 데이터가 마지막인 경우 해당 과거 데이터가 현재 배치 결과에 섞여 들어오는 "데이터 오염" 현상이 발생함.
- **해결**: `tail(1)` 대신 배치 전체 결과 중 가장 최신 시각(`max(occur_date)`)인 행들만 필터링하도록 수정하여, 현재 추론 주기에 해당하는 결과만 정확히 추출함.
